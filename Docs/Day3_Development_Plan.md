# 《丧尸生存 FPS》项目文档

## 第三天

## 一、今日开发计划

第三天的目标是进入完整战斗闭环：敌人能够生成后主动寻找玩家、靠近后攻击玩家，玩家能够使用武器对敌人造成伤害，并且伤害可以根据命中部位产生不同结果。同时完成第一版场景素材导入、NavMesh 寻路区域配置和基础烘焙渲染，为后续波次、掉落、祝福和关卡节奏打基础。

### 1.1 敌人系统架构设计

今日优先搭建敌人系统的基础架构，保证后续普通丧尸、精英敌人、远程敌人和飞行敌人可以在同一套框架下扩展。

规划敌人结构：
- EnemyController：敌人入口，负责状态机、组件引用和生命周期管理
- EnemyConfigAsset：敌人基础配置，负责生命值、移动速度、攻击力、攻击距离、攻击间隔、感知距离等静态数据
- EnemyRuntimeData：敌人运行时数据，负责当前生命、当前目标、是否死亡、是否正在攻击等局内状态
- EnemyStateMachine：敌人状态机，负责 Idle、Chase、Attack、Hit、Dead 等状态切换
- EnemyMotor：敌人移动模块，负责 NavMeshAgent 寻路、转向和移动控制
- EnemyHealth：敌人生命模块，负责受伤、死亡和部位伤害结算
- EnemyAttack：敌人攻击模块，负责攻击判定、攻击冷却和伤害触发
- EnemyHitbox：敌人部位受击模块，负责头部、身体、四肢等命中区域的识别
- EnemyView：敌人表现层，负责 Animator、受击反馈、死亡表现和特效挂点

设计原则：
- 敌人 AI 行为通过状态机管理，不把追击、攻击、死亡逻辑全部堆在 EnemyController 中
- 敌人移动统一通过 EnemyMotor 控制，便于后续切换地面敌人、飞行敌人或特殊移动方式
- 敌人生命和部位受击分离，武器命中时先命中 EnemyHitbox，再由 EnemyHealth 统一结算伤害
- 敌人配置和运行时数据分离，避免运行时修改污染基础配置

### 1.2 敌人寻路功能

完成第一版地面近战丧尸寻路功能，使敌人能够在场景中自动追踪玩家。

今日目标：
- 给场景配置 Navigation Static 和可行走区域
- 烘焙第一版 NavMesh
- 敌人预制体接入 NavMeshAgent
- 敌人生成后自动寻找玩家目标
- 玩家进入感知范围后，敌人进入 Chase 状态
- 敌人在 Chase 状态下持续 SetDestination 到玩家位置
- 敌人接近攻击距离后停止追击并进入 Attack 状态
- 玩家离开攻击距离后，敌人重新进入 Chase 状态

实现要求：
- 寻路逻辑只依赖玩家 Transform，不直接依赖玩家输入或 UI
- EnemyMotor 负责 NavMeshAgent 的速度、停止距离和路径刷新
- 路径刷新不必每帧执行，可以按固定间隔更新，减少性能浪费
- 敌人死亡后必须停止 NavMeshAgent，避免死亡模型继续滑动

### 1.3 敌人攻击功能

实现第一版近战攻击，让丧尸靠近玩家后可以造成伤害。

今日目标：
- 增加 EnemyAttack 模块
- 敌人进入攻击距离后触发攻击动画
- 通过攻击冷却控制攻击频率
- 攻击命中时调用玩家受伤接口
- 玩家死亡后敌人停止继续攻击
- 攻击范围和攻击伤害从 EnemyConfigAsset 中读取

攻击判定方案：
- 第一版采用距离判定，敌人攻击触发时检查玩家是否仍在攻击范围内
- 后续可升级为动画事件 + 碰撞盒判定，提高攻击表现和准确度

设计说明：
- Attack 状态只负责攻击节奏和状态切换
- 真正造成伤害由 EnemyAttack 执行
- 玩家扣血仍由 PlayerHealth 或玩家生命模块统一处理

### 1.4 敌人受伤与部位伤害

实现敌人按不同身体部位受伤，支撑爆头、身体伤害、四肢低伤害等基础战斗反馈。

部位规划：
- Head：头部，伤害倍率最高，用于爆头击杀和后续爆头祝福
- Body：身体，标准伤害
- Arm：手臂，较低伤害
- Leg：腿部，较低伤害，后续可扩展减速效果

今日目标：
- 给敌人预制体添加多个 EnemyHitbox
- 每个 EnemyHitbox 配置 BodyPartType
- 武器 Raycast 命中 EnemyHitbox 后，把命中部位传给 EnemyHealth
- EnemyHealth 根据部位倍率计算最终伤害
- 敌人受伤后播放受击反馈
- 生命值小于等于 0 时进入死亡流程
- 爆头击杀可以输出单独日志或事件，为后续祝福系统预留接口

伤害结算示例：

```text
最终伤害 = 武器基础伤害 * 武器倍率 * 命中部位倍率 * 当前局临时修正
```

第一版部位倍率建议：
- Head：2.0
- Body：1.0
- Arm：0.7
- Leg：0.6

设计说明：
- 武器系统不直接判断敌人死亡，只负责把命中信息和基础伤害传给敌人
- 敌人系统内部负责扣血、死亡、掉落和击杀事件
- 部位伤害接口要预留 DamageInfo，方便后续接入暴击、元素伤害、击退、燃烧、闪电等效果

### 1.5 武器系统开火功能完善

在第二天已有手枪、步枪、开镜、切枪基础上，继续完善武器开火对敌人的真实伤害闭环。

今日目标：
- 武器开火 Raycast 命中 EnemyHitbox 后造成伤害
- 命中普通场景物体时只播放命中特效或输出命中信息
- 命中敌人不同部位时使用不同伤害倍率
- 半自动武器和全自动武器继续保持现有输入规则
- 开火时同步触发枪口火光、命中特效、后坐力和摄像机反馈
- 子弹不足时不能开火
- 换弹、切枪、开镜状态下的开火限制保持正确

今日重点：
- 把“射线命中日志”升级为“真实命中敌人并扣血”
- 把武器系统和敌人系统通过 IDamageable 或 EnemyHitbox 解耦
- 为后续 HUD 命中提示、击杀提示、祝福能量获取预留事件

### 1.6 敌人死亡与击杀事件

完成敌人死亡流程的第一版实现。

今日目标：
- 敌人生命归零后进入 Dead 状态
- 停止寻路、停止攻击、关闭主要碰撞和受击盒
- 播放死亡动画或临时死亡表现
- 延迟回收或销毁敌人对象
- 发送 EnemyKilled 事件
- 事件中携带敌人类型、击杀部位、是否爆头、击杀者等信息

预留用途：
- 后续击杀敌人获得祝福能量
- 后续死亡后掉落道具
- 后续统计击杀数和金币结算
- 后续爆头击杀触发爆炸、闪电、印记等祝福效果

### 1.7 场景素材导入与初步烘焙

导入第一版场景素材，完成可测试的战斗区域。

今日目标：
- 导入或整理当前可用场景素材
- 搭建第一版测试地图
- 设置玩家出生点
- 设置敌人测试出生点
- 设置基础遮挡物、地面、高低差或通道
- 配置场景碰撞
- 配置 Navigation Static
- 烘焙 NavMesh
- 完成第一版灯光烘焙或基础光照设置
- 检查手机端运行时是否存在明显过暗、过曝、穿模或寻路失败问题

场景要求：
- 地图不追求最终美术效果，优先满足战斗测试
- 玩家要有足够空间移动、后退和拉扯敌人
- 敌人出生点不能离玩家过近，避免刚进场立即被攻击
- 地面和障碍物必须能正确参与寻路和碰撞
- 烘焙后需要实测敌人能从出生点追到玩家

### 1.8 今日测试清单

编辑器测试：
- 玩家移动、视角、开火、换弹、开镜、切枪正常
- 敌人可以自动追踪玩家
- 敌人接近玩家后可以攻击
- 玩家被敌人攻击后生命值减少
- 玩家武器命中敌人身体后敌人扣血
- 命中头部时伤害高于身体
- 命中手臂和腿部时伤害低于身体
- 敌人生命归零后死亡并停止移动
- 敌人死亡后不会继续攻击玩家
- 场景 NavMesh 没有明显断层或敌人卡死

真机测试：
- Android 包能正常启动
- 手机端移动、视角和开火仍然正常
- 开火命中敌人可以造成伤害
- 多个敌人同时追击时帧率无明显异常
- 场景光照在手机上显示正常
- 敌人寻路、攻击和死亡没有明显报错

### 1.9 第三日目标总结

第三天的重点是从“玩家可以开火”推进到“玩家和敌人可以真正战斗”。今日完成后，项目应具备第一版 FPS 战斗闭环：玩家进入场景，敌人通过 NavMesh 追击玩家，靠近后攻击玩家；玩家使用手枪或步枪开火，命中敌人不同部位产生不同伤害，敌人受伤、死亡并发送击杀事件。

本日完成后，后续第四天可以继续推进波次刷怪、敌人生成、击杀能量、道具掉落和肉鸽祝福选择。
